Grounding Context

Codebase Context

  • 产品形态： Snow CLI 上的自主编排插件（v0.2.0）：MCP 状态机 + 4 lifecycle hooks + 18 sub-agents + 10 skills + 18 commands。
  • 控制平面：planning → executing → verifying → done；文件写受 stage 门禁；onStop 驱动续推、text-bypass、auto-verify。
  • 已落地的完成门（L1）：gatesRequired + ledger 多闸（task-complete / task-reconcile / code-quality / completion）；成绩单 scorecard；reject → executing + lastGateFailure；禁止主身份自签 completion；写盘护栏拦 oms-state 旁路。
  • 仍明确 Deferred：完成门 L2（真独立 critic 会话 / nonce / teammate-only submit / worktree）。
  • 韧性资产： turn soft/hard cap、state 过期、atomic write、oms-state skill store、snapshot、doctor。
  • Team：/oms:team 已有 worktree 隔离与 lead 编排文案，但合并侧可观测/空结果过滤/嫁接综合仍偏提示词。
  • 设计语言：OMS-DESIGN-CONSTRAINTS.md — PRD、审批门、反伪造 token、阶段状态机、oms-state、file:line 证据。

External Context（workflow-cookbook）

  • 第 23 章：社区系统用「提示词 + Hook + 磁盘」模拟确定性编排；可迁移的是模式，不是宿主实现。
  • 第 24 章精华提取：解构→抽象→适配→验证；验证门循环、持久循环+hitCeiling、状态句柄、角色隔离、Category→Model。
  • 第 24.7 质量三拍：① 独立多视角 → ② 对抗验证 → ③ 代码做确定性收口（计票/去重/过滤交给代码，不交给模型）。
  • 附录 F 经典形状：Pipeline / Barrier / Adversarial / Judge Panel / Loop-until-X / Nested；场景→模式速查。
  • 第 25 章库化：具名 + 参数化 + 版本 + fixture 回归 + 自解释 meta（whenToUse）。
  • 第 26 章反模式：无界 while、预算不主动刹车、schema 过严、静默丢结果、控制流屏障误用、靠脚本主体写盘传数据。
与 Claude Code Workflow 的边界（不变）： OMS 跑在 Snow CLI，没有原生 agent()/pipeline()/schema 运行时。可借鉴控制结构与产品契约；用 MCP + hooks + skill/command 表达等价结构。

Ranked Ideas · Wave 2（门禁之后）

#1 结构化面包屑 #2 hitCeiling 诚实收口 #3 有界 GCF 金牌配方 #4 完成门 L2 隔离 #5 Loop-until-dry 发现配方 #6 评委/多透镜 plan #7 配方库 + 回归

1 结构化面包屑状态句柄（每轮固定块 + journal）

axis: A4 confidence: 90% complexity: Low–Medium was: 首版 #5

把 onStop / onUserMessage 注入从散文 guidance 升级为固定 schema 块：stage、turn soft/hard、tasks 剩余、last verify exit、PRD 进度、 gatesRequired、ledger 缺哪闸、lastGateFailure 摘要、open risks。可选 .snow/oms-state/journal.jsonl 追加「轮次 / 门事件 / 丢弃计数」供人机审计。

basis
direct: 门禁已写 ledger / lastGateFailure，但注入仍偏自然语言；压缩后易丢「下一闸该交什么」。
external: cookbook 第 24 章 ccg→状态句柄：显式、结构化传递；第 26 章 D3 截断必留痕。

rationale: 门禁把「能不能停」变成数据后，下一步最高杠杆是让主 agent 每轮都看见同一张控制面板——贴合现有 hook 架构，成本最低。

downsides: 注入过长挤占上下文；需硬上限 + 差分更新（只打变更字段）。

2 预算维度与诚实收口（hitCeiling）

axis: A1 confidence: 86% complexity: Medium was: 首版 #6

在 turn soft/hard cap 上增加「策略包」：接近 soft 时降级（少并行、跳可选 GCF）、触顶返回 hitCeiling:true + 未完成清单 + 已过哪些门，而不是假装 done 或死磕到 hard 崩溃。 用户可设 turn/分钟启发式预算（无原生 token 计数时诚实声明启发式）。

basis
direct: maxIterations/hardMaxIterations 已有，缺优雅降级与「撞顶 ≠ 成功」语义。
external: 第 18/21/26 章：有界 while + budget 主动刹车 + hitCeiling 标注；C1/C2 反模式。

rationale: 完成门堵住假绿后，第二大信任问题是「烧光预算还说差不多了」。诚实收口直接服务「自驾可信」叙事。

downsides: 无精确 token 时避免虚假精度；文案要区分「用户停」与「系统触顶」。

3 有界 GCF 金牌配方（Generate–Critique–Fix）

axis: A2/A5 confidence: 84% complexity: Medium was: 首版 #4

第一个可命名工序：独立 Critique 产出 issues[]（severity + file:line），Fix 逐条对账 changelog； 收敛 / 轮次帽（默认 3）/ 预算 三重刹车。可挂 cleanup、高风险路径、或 auto 可选增强；产出 issues 列表可再喂完成门 evidence。

basis
direct: cleanup + qa 循环存在，但缺「结构化挑刺列表 + 有界迭代」产品形状。
external: 第 12 章 GCF 实跑揪 10 缺陷；附录 F「有界 Loop + 对抗」；第 24 章 gatedFix 范本。

rationale: 门禁解决「能不能收工」；GCF 解决「中间质量工序可复现」——是配方库的样板工程。

downsides: 串行贵；Critique 无限挑刺必须轮次帽 + severity 过滤。

4 完成门 L2：真独立 critic 隔离

axis: A2 confidence: 80% complexity: High deferred from gates plan

把 L1「协议层禁止自签」升级为运行时隔离：nonce 仅 critic 可见、teammate-only submit、可选 worktree/独立会话。 材料协议：只给 claim + 原始证据（diff/stat、测试输出、task 列表），不给主 agent 辩护长文。

basis
direct: plan KTD3 / DoD 已诚实写明 L1 residual「伪 oms_critic 字符串」。
external: 第 17 章 refute-by-default；第 19 章 worktree；OmO VERIFICATION_REMINDER 精神。

rationale: 对安全/发布类会话，L1 只是止疼；L2 才是「对抗验证」的宿主级等价。

downsides: 依赖 Snow team/会话能力；实现与测试面大——应在 L1 dogfood 稳定后再开。

5 Loop-until-dry 发现配方（猎手池 + dry-streak）

axis: A5/A1 confidence: 78% complexity: Medium–High

面向「不知道有多少问题」的场景（分支扫雷、迁移残留、文档死链）：固定或自繁殖 finder 池 → 每条发现轻量对抗/去重 → 连续 K 轮无新增才停 + 硬轮次 + 预算。提供 lite（固定池）/ full（自繁殖）两档。

basis
direct: trace/dive/cleanup 偏单次深挖；缺「未知规模发现」的停止判据产品化。
external: 附录 F Loop-until-X；第 15/18/24.7 bughunt 骨架；dry-streak ≠ 一轮空就收。

rationale: 补齐「验收循环（已有门）」之外的另一半：发现循环。与 GCF 正交——一个找未知，一个修已知。

downsides: 成本上界需档位；误报需轻量 verify，避免每条五票（成本不匹配）。

6 多透镜草案 + 评委面板（强化 /oms:plan）

axis: A3/A5 confidence: 74% complexity: Medium

在 plan 路径可选：4 透镜（MVP / risk / dependency / user）独立草案 → M 评委打分 → 代码计票选冠军 → agent 只做「嫁接亚军亮点」的综合。确定性操作（计票、去重）不交给模型。

basis
direct: /oms:plan / plan skill 已有战略规划，但默认单路径生成。
external: 第 14 章 judge panel；第 24.7 plan-hunter；质量三拍 ①+②+③。

rationale: 解空间宽的规划任务最吃「独立多视角」；可显著拉开与「一上来就写一个 plan」的差距。

downsides: N×M agent 成本；仅大决策默认开启，小任务保持单路径。

7 具名配方库 + fixture 回归（产品化）

axis: A5 confidence: 72% complexity: High was: 首版 #7

把散落 command prompt 提升为版本化配方：name / description / whenToUse / phases / args 契约 / 退出条件 / 推荐拓扑 / _fixtures + 不变量测试。首批金牌：gcf、adversarial-done（已部分内建）、loop-until-dry、multidim-review、plan-panel。

basis
direct: 18 commands 多为长 prompt；skill 深但缺「库 + 回归 + 分享」层。
external: 第 25 章全章；附录 F 场景地图；成品清单 6 要件。

rationale: 平台叙事从「一个 auto」升级为「编排平台 + 可复用工序」；但应在 2–3 个金牌配方跑通后再铺库脚手架。

downsides: 文档与维护负担；过早抽象会空转——先配方后库。

Also Considered · Rejected

把 Claude Code Dynamic Workflow 运行时移植进 OMS — 宿主与协议不同；抽模式不抄实现。
无界 boulder / 只靠模型 done 退出 — 反模式 C1；与 hardMax + 完成门方向冲突。
重做 maturity Top-10 / 再立项一遍 L1 完成门 — 已在代码与 plan 中；重复会掩盖 Wave 2 机会。
默认对所有改动上 5 票陪审团 — 成本应匹配判错代价；仅高代价门/安全场景。
在 hook 主体靠写盘传 agent 间业务数据 — D2：跨 agent 走 MCP 状态/结构化返回，不靠隐式文件副作用（状态文件本身作为控制平面除外）。
Team 改成无协调自由并行 — 无屏障合并爆炸；独立任务才 pipeline，跨任务合并仍要 barrier。
用更严 schema「逼模型想深一点」 — D1：schema 管形状；深度靠 rubric/独立 critic，过严只会空转重试。
