1 有界 GCF 金牌配方（Generate–Critique–Fix）

axis: A2 / A5 confidence: High complexity: M was: Wave3 #1 · Wave2 #3

第一个可卖的工序：独立 Critique 产出 issues[]（severity + file:line + severity），Fix 逐条 changelog 对账； 收敛 / 默认 3 轮帽 / 预算或 turn 帽三重刹车。挂 cleanup、高风险路径、auto 可选增强；issues 列表可喂完成门 evidence。

basis
direct: cleanup/qa 循环存在但无「结构化挑刺列表 + 有界迭代」形状；完成门只卡收工不抬中间质量；v0.4 交接闭环后最大空洞仍是工序可复现性。
external: cookbook 第 12 章 GCF 实跑；附录 F「有界 Loop + 对抗」；第 24 章 gatedFix；C1/C2 反模式。

rationale: 「能不能收工 / 断了能否接」已闭环。下一跃迁是「中间质量可复现」——配方库样板，杠杆最高。

downsides / 取舍： 串行贵；Critique 无限挑刺必须轮次帽 + severity 过滤。 完整版做独立 skill+command；MVP 先嵌进 cleanup；最小可用只要求 issues schema + maxRounds=3。

2 Journal 审计轨（非静默截断）

axis: A4 confidence: High complexity: S–M promoted · was Wave3 #7

STATUS 解决「当下看见」；journal 解决「事后复盘 + 交接/合并/门事件留痕」。追加 .snow/oms-state/journal.jsonl：轮次、soft 续命、门 submit/reject、text-bypass、硬停、handoff 写/消费、team 丢弃/截断计数。 人读 + 可被 learn/handoff 消费；默认长度轮转。

basis
direct: plan 003/005 明确 Non-Goal journal；v0.4 交班与门事件已发生但无统一时间线；team 黑箱缺计数落点。
external: 第 26 章 D3 截断必留痕；第 28 章 journal 排错精神；质量三拍③ 确定性收口要可观测。

rationale: 交班已闭环后，下一低成本高贴现底座：放大门禁/面板/handoff/team 全部资产，避免再各写一套日志。

downsides / 取舍： 写盘与隐私 → 轮转 + 脱敏，禁止把 journal 当第二状态机。 MVP 只记事件类型+时间+关键计数；完整版再挂 query 工具。

3 Loop-until-dry 发现配方（猎手池 + dry-streak）

axis: A5 / A1 confidence: High complexity: M–H was: Wave3 #3

面向「不知道有多少问题」：固定或自繁殖 finder 池 → 轻量对抗/去重 → 连续 K 轮无新增才停 + 硬轮次 + turn/预算帽。lite（固定池）/ full（自繁殖）两档。与 GCF 正交：一个找未知，一个修已知。

basis
direct: trace/dive/cleanup 偏单次深挖；缺「未知规模发现」的停止判据产品化。
external: 第 15/18 章 bughunt + Loop-until-X；C1 禁止只靠模型 done；附录 F 场景表。

rationale: 补齐验收循环（门）之外的发现循环；适合迁移/扫雷/文档死链。

downsides / 取舍： 成本上界要档位；误报用轻量 verify，避免每条五票。先 lite 再 full。

4 Team 合并屏障：可观测 + 确定性过滤

axis: A3 / A4 confidence: High complexity: S–M was: Wave3 #4

把 team 合并从「提示词希望你 filter」升级为产品契约：合并前丢弃空/失败 teammate 结果并计数； 返回 { kept, dropped, reasons[] }；失败 teammate 不静默吞；可选「仅合并有 diff 的 worktree」。 确定性部分用代码/MCP，不交给模型计票；计数写入 journal/STATUS。

basis
direct: /oms:team 编排长、合并语义靠 lead 自觉；worktree 贵却无「合并质量」反馈。
external: 第 26 章 A3（先 filter）与 D3（截断必留痕）；第 19 章 worktree 成本纪律；第 24.7 ③。

rationale: 成本相对低、贴现有 team 路径、立刻抬「多 agent 可信度」；与 #2 journal 字段共用。

downsides / 取舍： 依赖 Snow team API 能否暴露失败/空结果；否则只能 lead 侧启发式 + 强制报告模板。

5 分片 / 多维 Review 工序（Scan→Review→Verify→Synthesize）

axis: A3 / A5 confidence: Med–High complexity: M was: Wave3 #5

「PR/大 diff 审查」具名工序：分片扫描 → 多维并行（正确性/安全/可维护/测试）→ 轻量对抗 → 代码侧去重与排序 → 综合报告。可作 /oms:verify 增强或独立 /oms:review；对接 code-quality scorecard evidence。

basis
direct: 有 oms_reviewer/oms_critic 与 completion 签核，缺大变更结构化审查流水线。
external: 第 10 章分片审查、第 11 章多维 PR review。

rationale: 完成门要「有人签过」；用户还要「签得有内容」。把 L1 门从形式门槛变成质量工序入口。

downsides / 取舍： 并行成本高；小 diff 单路径，大 diff 才分片。MVP 两维即可。

6 队友 Verify 真源（Lead-only 契约产品化）

axis: A3 / A2 confidence: Med–High complexity: M new in Wave 4

现状：teammate 写 .pending-verify 但永不跑 onStop，队友侧 auto-verify 是空承诺。 产品化选择其一并写死：（A）Lead-only 验证为唯一真源（STATUS 明示、禁止宣称 teammate 已测）； （B）合并后强制 lead 统一 verify 门 + 失败回执行；(C 完整版) 宿主支持 teammate onStop 再谈分布式 verify。 与 #4 合并可观测联动：无 diff / verify 失败的 teammate 不得静默并入。

basis
direct: assets/commands/oms/team.json 已写明 teammate auto-verify ineffective、lead-side onStop 为唯一真理——但仍是文案，非可执行契约与面板字段。
external: 第 24.7 确定性收口；D3 非静默；验证门循环（F.2）要求 pass 可机读。

rationale: 这是「多 agent 看起来在测、其实没测」的诚信漏洞；修它比再加一个 skill 更能守住 OMS 叙事。

downsides / 取舍： 不碰宿主协议则只能选 A/B；C 依赖 Snow。最稳妥先 A+B 产品化文案与门；C 进短名单。

7 嵌套一层：auto 内嵌金牌配方（仅一层）

axis: A5 confidence: Med complexity: M promoted from Wave3 shortlist

在 #1 GCF（及可选 #3 dry）存在后：/oms:auto 可选开关 --with gcf|dry|review 或 PRD 故事级「完成前跑一圈 GCF」。 嵌套仅一层：子配方不得再嵌子配方；共享 turn 预算与 STATUS 字段。

basis
direct: auto 是主入口，散装 skill 用户要自己串；learn 抽 skill 但不会挂到 auto 流水线。
external: 第 20 章 nested 仅一层；第 25 章具名库调用；附录 F nested 场景。

rationale: 金牌配方若不进 auto，永远是专家玩具；一层嵌套是「配方 → 产品默认路径」最短桥。

downsides / 取舍： 依赖 #1 先存在；过早嵌套会把未定型配方焊死。先 MVP：auto 结束前可选 GCF 一圈。
