1 结构化闸门契约（Schema-gated stage contracts）

axis: A2/A1 confidence: 88% complexity: Medium

在 stage 跃迁与关键 skill 收口点引入机器可读判决对象（JSON），而不是仅靠自然语言 continuation prompt。例如 verifying→done 需要 { pass, blockers[], evidence[] }；story mark-passes 已有 approval，可扩展到 auto/qa 的通用 gate record。

basis
direct: OMS 已有 PRD request-verification/submit-approval 与 stage 机，但对 general auto 路径，done 仍主要靠 agent 自报 + build/test。
external: cookbook 第 17/24 章：enum 三态 / pass:boolean 把门控从散文变成状态机迁移；schema 管形状、独立 agent 管真伪。

rationale: 这是 OMS 与「确定性骨架」差距最大、杠杆最高的一层：不换宿主也能把「过没过」变成可编程数据，支撑后续所有质量配方。

downsides: 需要定义最小 schema 与兼容策略；过严会触发反复重试式空转（反模式 D1）。

2 两段式验证阶段（Spec 合规 → 代码质量）

axis: A2 confidence: 84% complexity: Medium

把 verifying 从「跑测试 + 自由复盘」升级为固定两闸：① 对照 tasks/PRD/验收（有没有做对范围）② 独立 code quality/security 视角。每闸独立 sub-agent + gate record；第一闸不过不烧第二闸 token。

basis
direct: onStop 在 verifying 注入「review changes + build/test」；没有强制维度分离。
external: superpowers 两段式评审（cookbook 23/24）；GCF 与对抗验证的「生成与评估分离」。

rationale: 测试绿 ≠ 做对需求；两闸正交，直接抬升「自驾可靠」叙事可信度。

downsides: wall-clock 与 token 增加；小改动需要 skip/轻量模式。

3 done 前独立对抗证伪（refute-by-default）

axis: A2 confidence: 82% complexity: Medium

对「可以收工」的主张默认证伪：派独立 #oms_critic/#oms_reviewer，只给 claim + 原始证据（diff/stat、测试输出、task 列表），不给主 agent 的辩护长文；判决 confirmed|refuted|uncertain，默认不放行 uncertain。

basis
direct: text-bypass 只抓「声称改了但 git 无 diff」；Ralph 有 reviewer 签核，auto 路径较弱。
external: 第 17 章对抗验证；OmO VERIFICATION_REMINDER「子 agent 说完成了就去验证」。

rationale: 终结「自我评估完成」——这是 OMC/OMS 类系统最容易假绿的点。

downsides: 至少 +1 agent；误杀真完成时需要 rework 路径与置信度。

4 有界 GCF 配方（Generate–Critique–Fix）

axis: A2/A5 confidence: 80% complexity: Medium

新增可调用配方（skill 或 command）：独立 Critique 产出 issues[]，Fix 逐条对账并带 changelog；收敛 / 轮次上限（经验 3）/ budget 三重刹车。可挂在 high-risk 文件、cleanup、或 auto 对关键模块的可选增强。

basis
direct: 有 cleanup skill 与 qa 循环，但缺「结构化挑刺列表 + 有界迭代」产品形状。
external: 第 12 章 GCF 实跑揪出 10 缺陷；第 18/26 章禁止无界 while。

rationale: 把「写完自己瞅一眼」升级为可复现的质量工序，且能作为第一个「配方库」样板。

downsides: 串行三阶段贵；需防 Critique 无限挑刺（必须轮次帽）。

5 强化面包屑状态句柄（每轮结构化注入）

axis: A4 confidence: 78% complexity: Low–Medium

把 onStop/onUserMessage 注入从散文 guidance 升级为固定块：stage、turn/soft/hard、tasks 剩余、last verify exit、PRD 进度、open risks、last gate record 摘要。压缩后仍可读；可选写入 .snow/oms-state/journal.jsonl 供人机审计。

basis
direct: onStop 已注入 stage/PRD/turn 进度，但非统一 schema，journal 人读性弱。
external: ccg <ccg-state> 面包屑；第 24 章「显式、结构化传递」原则。

rationale: OMS 本就靠 hook 续命——这是在既有架构上成本最低的抗失忆升级。

downsides: 注入过长挤占上下文；需严格上限与差分更新。

6 预算维度与诚实收口（hitCeiling）

axis: A1 confidence: 76% complexity: Medium

在 turn cap 之外增加「成本/轮次策略包」：预估每轮代价、接近 soft max 时降级（少并行、跳可选 GCF）、触顶时返回 hitCeiling:true + 未完成清单，而不是假装 done。用户可设 token/分钟预算（若宿主可观测；否则用 turn×启发式）。

basis
direct: maxIterations/hardMaxIterations 已存在，缺「优雅降级」与诚实失败语义。
external: 第 18/21/26 章 budget 硬上限、ROUND_COST 主动刹车、hitCeiling 标注。

rationale: 自主循环的可信来自「停得住且说得清」，不是「跑得最久」。

downsides: 无原生 token 计数时只能启发式；产品文案要避免虚假精度。

7 具名配方库（Recipe library）产品化

axis: A5/A3 confidence: 74% complexity: High

把散落在 command prompt 里的流程提升为版本化配方：meta.name/description/phases、参数 schema、退出条件、推荐 agent 拓扑、回归用例（fixture 仓库 dogfood）。首批配方：sharded-review、gcf、adversarial-done、prd-acceptance-loop、team-pipeline。

basis
direct: 18 个 command 多为长 prompt 字符串；skill 深度在增强，但缺少「库 + 版本 + 分享」层。
external: 第 25 章 workflow 库；附录 F 场景→模式地图。

rationale: 把 OMS 从「一个通用 auto」升级为「编排平台 + 可复用工序」，形成与 OMC/cookbook 同级的生态语言。

downsides: 产品与文档负担大；应先做 2–3 个金牌配方再谈库。

Also Considered · Rejected

把 Claude Code Dynamic Workflow 运行时移植进 OMS — 宿主与协议完全不同；违反「抽模式不抄实现」。应在 Snow 能力边界内用 MCP+hooks 表达等价控制结构。
无界 boulder / 只靠模型 done 退出 — cookbook 反模式 C1；与现有 hardMax 方向冲突，会烧光用户预算。
复述 maturity Top-10（timeout/||/doctor/CI）为本 ideation 主线 — 已有 implementation-ready 计划；重复会掩盖「P0 之后」的真实机会。
默认对所有改动上 5 票陪审团 — 第 17 章明确成本应匹配判错代价；应仅用于 done/安全/发布类高代价门。
在 hook 脚本主体靠写盘传 agent 间数据 — cookbook D2：跨 agent 应用结构化返回值/状态工具，而不是隐式文件副作用。
Team 模式全面改成无协调自由并行 — 无屏障会在合并阶段爆炸；只在「任务独立」时采用 pipeline 语义，跨任务去重/合并仍要 barrier。

Suggested Pursuit Order

  1. 先契约后配方：#1 闸门记录 → 解锁 #2/#3 的可编程分支。
  2. 低成本韧性：#5 面包屑 + #6 诚实收口，贴合现有 hook 架构。
  3. 可卖点质量：#2 两段验证 + #3 对抗 done，直接抬升「不会假绿」。
  4. 可复制工序：#4 GCF 作为首个金牌配方 → 再铺 #7 库。

与 maturity PR 栈关系：建议 U1–U10 可信度落地后（或并行不冲突的文档/skill 层），再开闸门契约；避免在 verify 仍可能静默失败时堆更复杂质量门。
