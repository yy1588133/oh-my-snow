1 有界 GCF 金牌配方（Generate–Critique–Fix）

axis: A2 / A5 confidence: 90% complexity: Medium was: Wave2 #3

第一个可卖的工序：独立 Critique 产出 issues[]（severity + file:line + severity），Fix 逐条 changelog 对账； 收敛 / 默认 3 轮帽 / 预算三重刹车。可挂 cleanup、高风险路径、auto 可选增强；issues 列表可再喂完成门 evidence。

basis
direct: cleanup/qa 循环存在，但缺「结构化挑刺列表 + 有界迭代」产品形状；完成门只卡收工，不提升中间质量。
external: cookbook 第 12 章 GCF 实跑揪缺陷；附录 F「有界 Loop + 对抗」；第 24 章 gatedFix 范本。

rationale: v0.3.0 已解决「能不能收工、能不能看见」。下一跃迁是「中间质量工序可复现」——配方库的样板工程，杠杆最高。

downsides: 串行贵；Critique 无限挑刺必须轮次帽 + severity 过滤。

2 硬停交接包 / 跨会话续跑（Handoff Pack）

axis: A1 / A4 confidence: 86% complexity: Medium new in Wave 3

硬停完整体检已经「说清楚了」；下一步是把接手变成一键可续：落盘 handoff.json（goal、stage、tasks 剩余、缺门、lastGateFailure、verify/PRD 要点、建议下一刀）+ 可选 snapshot 指针。 提供 /oms:auto --resume 或 /oms:resume：读交接包 → 新会话从正确 stage/任务续，不假装继承同一上下文窗口。

basis
direct: 硬停文案已有；snapshot/save 存在但未与 hard ceiling 打成「标准交接」；用户撞顶后仍靠人脑拼凑。
external: cookbook 第 22 章明确：同会话 resumeFromRunId ≠ 跨会话；跨会话状态必须走磁盘产物。OMS 适配的是「控制面 handoff」，不是 agent 缓存。

rationale: 诚实停若不能「干净接着干」，信任会停在半路。这是把 v0.3.0 硬停资产变成完整闭环的最短路径。

downsides: 勿承诺「上下文无损续」；只承诺任务/门/产物可续。旧会话 state 清理策略要写清。

3 Loop-until-dry 发现配方（猎手池 + dry-streak）

axis: A5 / A1 confidence: 82% complexity: Medium–High was: Wave2 #5

面向「不知道有多少问题」：固定或自繁殖 finder 池 → 轻量对抗/去重 → 连续 K 轮无新增才停 + 硬轮次 + 预算。提供 lite（固定池）/ full（自繁殖）两档。与 GCF 正交：一个找未知，一个修已知。

basis
direct: trace/dive/cleanup 偏单次深挖；缺「未知规模发现」的停止判据产品化。
external: 第 15/18 章 bughunt + Loop-until-X；C1 反模式禁止只靠模型 done。

rationale: 补齐验收循环（已有门）之外的发现循环；适合迁移/扫雷/文档死链。

downsides: 成本上界要档位；误报只需轻量 verify，避免每条五票。

4 Team 合并屏障：可观测 + 确定性过滤

axis: A3 / A4 confidence: 80% complexity: Low–Medium promoted from Wave2 shortlist

把 team 合并从「提示词希望你 filter」升级为产品契约：合并前丢弃空/失败 teammate 结果并 log 计数；返回 { kept, dropped, reasons[] }；失败 teammate 不静默吞； 可选「仅合并有 diff 的 worktree」。确定性部分用代码/MCP，不交给模型计票。

basis
direct: /oms:team 编排长、合并语义靠 lead 自觉；黑箱是 dogfood 高发区。
external: 第 26 章 A3（parallel 结果先 filter）与 D3（截断必留痕）；第 24.7 质量三拍③。

rationale: 成本低、贴现有 team 路径、立刻抬「多 agent 可信度」；与 journal/STATUS 字段可共用。

downsides: 依赖 Snow team API 能暴露失败/空结果信号；否则只能做 lead 侧启发式。

5 分片 / 多维 Review 工序（Scan→Review→Verify→Synthesize）

axis: A3 / A5 confidence: 78% complexity: Medium new in Wave 3

把「PR/大 diff 审查」做成具名工序：分片扫描 → 多维并行（正确性/安全/可维护/测试）→ 轻量对抗核实 → 代码侧去重与排序 → 综合报告。可作为 /oms:verify 增强或独立 /oms:review；与完成门 code-quality 的 scorecard 可对接。

basis
direct: 有 oms_reviewer/oms_critic 与 completion 签核，但缺「大变更结构化审查流水线」。
external: 第 10 章分片审查、第 11 章多维 PR review 的真实配方形状。

rationale: 完成门要的是「有人签过」；用户还要「签得有内容」。分片 review 把 L1 门从形式门槛变成质量工序入口。

downsides: 并行成本高；小 diff 应默认单路径，大 diff 才开分片。

6 多透镜草案 + 评委面板（强化 /oms:plan）

axis: A3 / A5 confidence: 74% complexity: Medium was: Wave2 #6

plan 路径可选：4 透镜（MVP / risk / dependency / user）独立草案 → M 评委打分 → 代码计票选冠军 → agent 只做「嫁接亚军亮点」。确定性操作不交给模型。

basis
direct: /oms:plan 默认单路径生成。
external: 第 14 章 judge panel；第 24.7 plan-hunter；质量三拍 ①+②+③。

rationale: 解空间宽的规划最吃独立多视角；拉开「一上来就写一个 plan」的差距。

downsides: N×M 成本；仅大决策默认开，小任务保持单路径。

7 Journal 审计轨（非静默截断）

axis: A4 confidence: 72% complexity: Low–Medium residual of Wave2 #1

STATUS 解决「当下看见」；journal 解决「事后复盘」。追加 .snow/oms-state/journal.jsonl：轮次、soft 续命、门事件、text-bypass、丢弃/截断计数、硬停事件。 人读 + 可被 learn/handoff 消费。默认有长度轮转。

basis
direct: plan 003 明确 Non-Goal「本期不做 journal」；面板已有但无历史轨。
external: 第 26 章 D3 截断必留痕；第 28 章 journal 排错精神。

rationale: 低成本、放大已有门禁与面板资产；为 team 丢弃计数与 handoff 提供统一时间线。

downsides: 写盘与隐私；要轮转/脱敏，避免日志变第二状态机。

Also Considered · Rejected

把 Claude Code Dynamic Workflow 运行时移植进 OMS — 宿主与协议不同；抽模式不抄实现。
同会话 resumeFromRunId 级 agent 缓存克隆 — cookbook 第 22 章依赖 Workflow 脚本身份与运行时缓存；Snow 无此语义。跨会话走 handoff 磁盘产物即可。
无界 boulder / 只靠模型 done 退出 — 反模式 C1；与 hardMax + 完成门冲突。
再立项一遍 L1 完成门 / 状态面板 — v0.3.0 已落地；重复掩盖 Wave 3 机会。
默认对所有改动上 5 票陪审团 — 成本应匹配判错代价；仅高代价门/安全场景。
在 hook 主体靠写盘传 agent 间业务数据 — D2：跨 agent 走 MCP 状态/结构化返回（状态文件作为控制平面除外）。
Team 改成无协调自由并行 — 无屏障合并爆炸；独立任务才 pipeline，跨任务合并仍要 barrier。
用更严 schema「逼模型想深一点」 — D1：schema 管形状；深度靠 rubric/独立 critic。
软 cap 一到就停 — 与现已产品化的「软续命 + 硬停」契约冲突；降速可议，取消续命不议。
